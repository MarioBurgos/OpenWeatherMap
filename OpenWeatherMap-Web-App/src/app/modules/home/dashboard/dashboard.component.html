There you are!

I will start saying that I started this mini-project thinking on its modular structure. I wanted to code an app that is truly scalable and portable. That is why I created a module for each link at the navbar that contain the components that I believed were necessary.

Then, the structure I thought is the following:

Modular structure
Modules & Components


See that the page is built in a layout that contains the Navigation Bar and the Footer. The Skeleton, which is the frame-like of the page, has both of the former named components plus the router-outlet in it, making the whole Single Page App as it can be seen in the next image.

'skeleton.component.html'


The routing has been constructed in lazy-loading type for each module.

'app-routing.module.ts'
'weather-routing.module.ts'


By clicking on the navbar link 'Weather', the router loads the Search Component.

In order to pass the input text to the 'Current' and 'Forecast' Components for them to know which location data must they look for through the 'FetchWeatherService' (see the image called 'fetch-weather.service.ts' below), I took the quick and simple option. I created a 2-way binding between the input and an attribute called location and then I added the parameter to the url at the [routerLink] of each button.

'search.component.html'


Each button leads to two different components. Both of them will make an API call using the location typed, through a service:

'fetch-weather.service.ts'
(I know, I should have all these strings in a constants file... And the 'apiKey' somehow protected)


The first one component it is shown by clicking the Current Weather button, and brings information by making a call to the OpenWeatherMap API through the 'fetch-weather.service'.

See that the subscriptions are created 'ngOnInit' and they are dumped 'ngOnDestroy'.

'current.component.ts'


Well, once the service has fetch the data it is only a matter of web desigh to show it on the component. I know it can be done better than this, but I like it simple :D
